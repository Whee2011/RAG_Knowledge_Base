## 二、P1 高优先级问题修复验证 (6/6)

### 6. ✅ 已修复 — Embedding 错误返回全零向量
- **原问题**: `_embed()` 失败时返回 `[0.0] * 2560`
- **现状态**: 已改为 `raise` 抛出异常:
  ```python
  except Exception as e:
      print(f"[Error] Embedding 错误：{e}")
      raise
  ```
  同时新增了空 Embedding 检查:
  ```python
  if not embedding:
      raise Exception("LM Studio 返回空 Embedding")
  ```
- **注意事项**: `embed_documents()` 中循环调用 `_embed()`，如果某一条文本失败会中断整个批处理。这是正确行为（避免全零向量污染），但调用方应做好异常捕获。`add_documents()` 中已有 `try/except` 包裹。
- **结论**: 已正确修复

### 7. ✅ 已修复 — PDF OCR 路径大小写替换 bug
- **原问题**: `filepath.replace('.pdf', '_ocr.txt').replace('.PDF', '_ocr.txt')` 链式替换不安全
- **现状态**: 新增 `_get_ocr_txt_path()` 辅助方法:
  ```python
  def _get_ocr_txt_path(self, filepath: str) -> str:
      base, _ = os.path.splitext(filepath)
      return base + '_ocr.txt'
  ```
  `_scan_documents()` 和 `_auto_ocr()` 中均已使用此方法
- **结论**: 已正确修复

### 8. ✅ 已修复 — `_split_text` 无限循环风险
- **原问题**: `chunk_overlap >= chunk_size` 时 `start` 不前进导致死循环
- **现状态**: 添加了两个保护:
  ```python
  if self.chunk_overlap >= self.chunk_size:
      self.chunk_overlap = max(0, self.chunk_size // 4)
  # ...
  start = max(next_start, start + 1)  # 确保至少前进 1 字符
  ```
- **结论**: 已正确修复

### 9. ✅ 已修复 — Web API 路径遍历攻击
- **原问题**: `api_delete` 和 `api_ocr_file` 未验证路径合法性
- **现状态**: 新增 `secure_document_path()` 函数:
  ```python
  def secure_document_path(filepath: str) -> str:
      full_path = os.path.normpath(os.path.join(DOCUMENTS_PATH, filepath))
      normalized_doc_path = os.path.normpath(DOCUMENTS_PATH)
      if not full_path.startswith(normalized_doc_path):
          raise ValueError('非法路径')
      return full_path
  ```
  `api_delete` 和 `api_ocr_file` 均使用此函数验证路径
- **遗留风险**: `startswith` 检查在 Windows 上可能存在大小写敏感问题。建议改为 `os.path.normpath(full_path).lower().startswith(normalized_doc_path.lower())` 以增强健壮性。
- **结论**: 修复到位，有轻微改进空间

### 10. ✅ 已修复 — `/api/upload` 无文件大小限制
- **原问题**: 无上传文件大小限制
- **现状态**: 已设置 `app.config['MAX_CONTENT_LENGTH'] = 500 * 1024 * 1024` (500MB)
- **结论**: 已正确修复。500MB 对于文档上传场景较为宽松但合理。

### 11. ✅ 已修复 — Excel 数值类型转换 bug
- **原问题**: `float(value) if '.' in str(value) else int(value)` 不处理千分位逗号
- **现状态**: 已改为:
  ```python
  value = pd.to_numeric(str(value).replace(',', ''), errors='coerce')
  if pd.isna(value):
      print(f"[Warning] 无法将 '{condition.get('value')}' 转换为数值")
      continue
  ```
- **结论**: 已正确修复

## 五、新发现的问题

### N1. [P1] `_embed()` 抛出异常后 `embed_documents()` 循环中断风险
**文件**: `F:\RAG_Knowledge_Base\core\core.py` 第 60-63 行
**问题**: `embed_documents()` 对文本列表逐条调用 `_embed()`。如果某一条文本的 Embedding 请求失败，整个批处理会中断，导致之前已成功嵌入的文本也被丢弃。在 `add_documents()` 中这会导致整个文档索引失败。
**建议**: 在 `embed_documents()` 中对单条失败做重试或跳过处理，而非直接传播异常导致整批失败。

### N2. [P2] `core_qdrant.py` 的 `_embed()` 仍然返回全零向量
**文件**: `F:\RAG_Knowledge_Base\core\core_qdrant.py` 第 97 行
```python
except Exception as e:
    print(f"[Error] Embedding 错误：{e}")
    return [0.0] * self.dimension  # ← 仍然是全零向量
```
**问题**: 原 P1 #6 的修复只应用于 `core.py`，但 `core_qdrant.py` 中的 `LMStudioEmbeddings._embed()` 仍然返回全零向量，存在同样的数据库污染风险。虽然 `core_qdrant.py` 是备用文件，但如果被使用则会触发此问题。
**建议**: 同步修复 `core_qdrant.py` 中的 `_embed()` 方法。

### N3. [P2] `secure_path` 在 Windows 上的大小写敏感性
**文件**: `F:\RAG_Knowledge_Base\web\web_app.py` `secure_document_path()` 函数
```python
if not full_path.startswith(normalized_doc_path):
```
**问题**: Windows 文件路径不区分大小写，但 `startswith` 做的是区分大小写的比较。攻击者可能通过大小写变换绕过检查（如 `DOCUMENTS\..\..\Windows` 的小写变体）。
**建议**: 改为不区分大小写比较:
```python
if not full_path.lower().startswith(normalized_doc_path.lower()):
```

### N4. [P2] `core_qdrant.py` `_chunk_text` 仍然缺少无限循环保护
**文件**: `F:\RAG_Knowledge_Base\core\core_qdrant.py`
**问题**: `core.py` 中的 `_split_text` 已修复（添加了 `chunk_overlap >= chunk_size` 自动修正和 `start + 1` 保护），但 `core_qdrant.py` 中的 `_chunk_text` 仍然使用:
```python
start = end - self.chunk_overlap
```
缺少无限循环保护。

### N5. [P3] `core_chromadb.py` 未同步修复
**问题**: `core_chromadb.py` 是一个独立副本，未应用以下修复:
- 仍然使用 `.replace('.pdf', '_ocr.txt').replace('.PDF', '_ocr.txt')` 路径替换（P1 #7）
- `_embed()` 仍返回全零向量（与 N2 类似，但这是 `core_chromadb.py` 的副本）
- `_detect_need_ocr` 仍只检查第一页
**建议**: 如果 `core_chromadb.py` 继续保留，应从 `core.py` 同步这些修复；或者明确标注为废弃文件。
